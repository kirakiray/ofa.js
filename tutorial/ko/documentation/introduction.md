# 소개

## ofa.js란 무엇인가?

ofa.js는 사용하기 쉬운 웹 프론트엔드 프레임워크로, 웹 애플리케이션 구축을 위해 설계되었으며, 개발 효율성을 높이고 개발 진입 장벽을 낮추는 것을 목표로 합니다.

> HTML, CSS 및 JavaScript에 **이미 기본적으로 익숙하다면**, ofa.js는 매우 좋은 선택입니다.

## 내가 왜 ofa.js를 개발했나

ofa.js의 설계 초심은 간단합니다: 컴파일 도구에 의존하지 않고 웹 개발의 **엔지니어링** 문제를 해결하는 것입니다.

> 공학화는 소프트웨어 개발 과정의 규범, 프로세스, 도구 및 방법을 체계적으로 통합하여 개발 효율성, 코드 품질 및 유지보수성을 향상시키는 것을 의미합니다.

최근 10년 동안 Web 프론트엔드 개발은 초기 무질서한 성장에서 점차 엔지니어링화로 진화하는 과정을 겪었습니다. 이 과정은 전통적인 애플리케이션 개발의 경험을 참고하여 Node.js와 컴파일 플로우를 도입함으로써 대규모 프로젝트의 엔지니어링 문제를 해결했습니다.

그러나 프로젝트가 커지면서 이러한 패턴의 단점이 점차 드러나며, 전통적인 개발에서 직면하는 **거대한 애플리케이션** 문제가 발생하여 프로젝트 유지보수가 어려워지고 상호작용 요구사항을 반복적으로 업데이트하기가 어려워졌습니다.

> 모놀리식 애플리케이션(Monolithic Application)은 하나의 방대하고 밀접하게 결합된 단일 애플리케이션으로, 모든 기능 모듈이 하나의 코드베이스에 집중되어 있어 하나를 건드리면 전체가 영향받으며, 독립적으로 개발하고 배포하기 어렵습니다.

이때는 프로젝트를 분해하여 마이크로 프론트엔드화를 구현해야 합니다, **마이크로서비스**처럼요. 그러나 컴파일 처리를 거치면서 마이크로 프론트엔드는 어렵고 번거로워집니다. 프론트엔드 모듈의 독립적인 배포는 각각의 작은 모듈을 컴파일해야 하므로 매우 어렵습니다. 이로 인해 웹 프론트엔드 기술의 발전이 거의 정체 상태에 놓이게 되었습니다.

> 마이크로서비스(Microservices)는 소프트웨어 아키텍처 스타일로, 대규모의 복잡한 애플리케이션을 여러 개의 세분화되고 독립적으로 배포 및 실행되는 소형 서비스로 분할합니다.

이때, 나는 생각하기 시작했다: 전통적인 프로그래밍 언어는 서로 다른 하드웨어와 운영 체제에 대응해야 하기 때문에 컴파일을 통해 플랫폼 간 차이를 없애야 한다. 하지만 웹 개발은 다르다. 브라우저를 기반으로 하기 때문에 원래 컴파일이 필요 없고 독립적으로 실행 및 배포할 수 있으며, 본래 마이크로 프론트엔드 방식이다. 그래서 나는 컴파일 과정이 오히려 일을 복잡하게 만든다는 것을 깨달았다.

즉, 공학화 문제를 해결하고 컴파일이라는 필수 단계를 없애면 프론트엔드 개발이 대규모 애플리케이션 개발에 매우 적합해지며, 이것이 바로 타고난 마이크로 프론트엔드 패턴입니다. 그래서 ofa.js가 탄생했습니다.

## 핵심 강점

### 진입 장벽 제로, 개봉 즉시 사용 가능

개발 환경을 구축하거나 종속성을 설치하거나 스캐폴딩을 구성할 필요 없이 브라우저에서 공식 빌드 프로그램을 열고 로컬 디렉터리를 선택하면 개발을 시작할 수 있습니다. 모든 계산, 데이터 및 저장 작업은 로컬에서 실행되며 클라우드 서비스에 의존할 필요가 없습니다.

### AI 친화적, 검증 용이

컴파일 블랙박스 없이, AI가 생성한 코드는 빠르게 배포하고 자체 검증할 수 있습니다; 중간 계층을 줄이고 컴파일 과정을 거치지 않음으로써 코드의 문제를 더 쉽게 찾고 수정할 수 있습니다.

### 마이크로 프론트엔드 네이티브 지원

ofa.js는 웹 프론트엔드 개발을 마이크로서비스처럼 여러 개의 독립적인 모듈로 세분화하여 각 모듈을 독립적으로 개발하고 배포할 수 있게 합니다. 전통적인 웹 프론트엔드의 경계가 허물어지면 프론트엔드 기술은 점차 서버 기술의 한계를 돌파할 것입니다.
