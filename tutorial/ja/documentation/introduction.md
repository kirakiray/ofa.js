# 紹介

## ofa.jsとは何ですか？

ofa.js は使いやすい Web フロントエンドフレームワークで、Web アプリケーションの構築に特化して設計されており、開発効率を向上させ、開発のハードルを下げることを目的としています。

> HTML、CSS、JavaScript に**すでに基本的に精通している**なら、ofa.js は非常に良い選択肢です。

## なぜ私が ofa.js を開発したのか

ofa.js の設計意図は非常にシンプルです：コンパイルツールに依存せずに、Web開発の**エンジニアリング**問題を解決することです。

> エンジニアリングとは、ソフトウェア開発プロセスにおける規範、フロー、ツール、方法を体系的に統合し、開発効率、コード品質、保守性を向上させることを指します。

ここ10年、Webフロントエンド開発は初期の無秩序な成長から、次第にエンジニアリング化へと進化してきました。この過程では、従来のアプリケーション開発の経験を参考にし、Node.jsとコンパイルプロセスを導入することで、大規模プロジェクトのエンジニアリング上の課題を解決してきました。

しかし、プロジェクトが大きくなるにつれて、このパターンの欠点が徐々に明らかになり、従来の開発で直面していた**モノリスアプリケーション**の問題が発生し、プロジェクトのメンテナンスが困難になり、インタラクション要件の反復が難しくなりました。

> 巨石アプリケーション（Monolithic Application）とは、庞大で密結合な单体アプリケーションであり、すべての機能モジュールが1つのコードベースに集約されており、一部分が变动すれば全体に影響し、独立した開発やデプロイが難しい。

この時点で、プロジェクトを分解し、**マイクロサービス**のようにマイクロフロントエンド化を実現する必要があります。しかし、コンパイル処理を経ているため、マイクロフロントエンドは困難で煩雑になり、フロントエンドモジュールの独立したデプロイには各小さなモジュールのコンパイルが必要で、これは非常に困難であり、Webフロントエンド技術の発展はほぼ停滞に直面しています。

> マイクロサービス（Microservices）は、大規模で複雑なアプリケーションを、細かい粒度で独立してデプロイ・実行可能な複数の小さなサービスに分割するソフトウェアアーキテクチャスタイルです。

この時、私は考え始めました：従来のプログラミング言語は異なるハードウェアやオペレーティングシステムに対応する必要があるため、コンパイルを通じてクロスプラットフォームでの差異をなくさなければなりません。しかし、Web開発は異なります。ブラウザをベースにしており、もともとコンパイルが不要で、独立して実行およびデプロイでき、生まれつきマイクロフロントエンドのモードです。そこで私は、コンパイルのプロセスがかえって物事を複雑にしていることに気づきました。

つまり、エンジニアリングの問題を解決し、コンパイルという必要なステップを取り除けば、フロントエンド開発は大規模アプリケーションの開発に非常に適しているということです。これが生まれつきのマイクロフロントエンドモードです。そこでofa.jsが誕生しました。

## コアアドバンテージ

### 敷居ゼロ、箱から出してすぐ使える

開発環境の構築、依存関係のインストール、またはスキャフォールドの設定は不要です。ブラウザで公式ビルドプログラムを開き、ローカルディレクトリを選択するだけで開発を開始できます。すべての計算、データ、およびストレージ操作はローカルで実行され、クラウドサービスに依存する必要はありません。

### AI フレンドリーで検証しやすい

コンパイルのブラックボックスがなく、AIが生成したコードは迅速にデプロイと自己検証が可能です。中間層を減らし、コンパイルプロセスを経ずに、コードの問題の特定と修正を容易にします。

### マイクロフロントエンドのネイティブサポート

ofa.js は、Webフロントエンド開発をマイクロサービスのように細分化し、各モジュールを独立して開発・デプロイ可能にします。従来のWebフロントエンドの境界が打破されると、フロントエンド技術は徐々にサーバー技術の制限を突破していくでしょう。
