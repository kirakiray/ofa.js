# 紹介

## ofa.jsとは？

ofa.js は使いやすい Web フロントエンドフレームワークで、Web アプリケーションの構築に特化し、開発効率の向上と開発のハードルを下げることを目的としています。

> HTML、CSS、JavaScript **すでに基本的に理解している** なら、ofa.js は非常に良い選択です。

## なぜ私がofa.jsを開発したのか

ofa.jsの設計意図は簡単です：コンパイルツールに依存せずに、Web開発の**エンジニアリング**問題を解決することです。

> エンジニアリングとは、ソフトウェア開発プロセスにおける規範、フロー、ツール、方法を体系的に統合し、開発効率、コード品質、および保守性を向上させることを指します。

この十年間、Web フロントエンド開発は初期の乱開発から徐々にエンジニアリング化へと進化する過程を経てきました。この過程では、従来のアプリケーション開発の経験を参考に、Node.js とコンパイルフローの導入により、大規模プロジェクトのエンジニアリング上の課題を解決しました。

しかし、プロジェクトが大きくなると、このパターンの欠点が徐々に露呈し、従来の開発で直面する**モノリシックアプリケーション**の問題が発生し、プロジェクトの保守が難しく、インタラクション要件の反復が困難になります。

> モノリシックアプリケーション（Monolithic Application）とは、巨大で緊密に結合されたモノリシックなアプリケーションを指し、すべての機能モジュールが1つのコードベースに集中しており、一部を変更すると全体に影響を及ぼし、独立した開発やデプロイが困難です。

この時、プロジェクトを分解し、マイクロフロントエンド化を実現する必要があります。**マイクロサービス**のように。しかし、コンパイル処理を経ているため、マイクロフロントエンドは困難で煩雑になり、フロントエンドモジュールの独立したデプロイには各小さなモジュールをコンパイルする必要があり、これは非常に困難であり、Webフロントエンド技術の発展はほぼ停滞しています。

> マイクロサービス（Microservices）はソフトウェアアーキテクチャスタイルの一つで、大規模で複雑なアプリケーションを複数の細粒度の、独立してデプロイ・実行可能な小型サービスに分割します。

このとき、私は考え始めた。従来のプログラミング言語は異なるハードウェアやOSに対応しなければならないため、コンパイルによってクロスプラットフォームの差異を解消する必要がある。しかしWeb開発はブラウザを基盤にしており、もともとコンパイル不要で独立して動作・デプロイでき、生来的にマイクロフロントエンドのモデルである。そこで私は気づいた、コンパイルのプロセスがむしろ事を複雑にしていたのだ。

つまり、エンジニアリングの問題を解決し、コンパイルという必須のステップを排除すれば、フロントエンド開発は大規模アプリケーションの開発に非常に適しており、これは生まれつきのマイクロフロントエンドパターンです。そこで ofa.js が登場しました。

## 核心的な強み

### ゼロハードル、すぐに使える

開発環境の構築や依存関係のインストール、スキャフォールディングの設定は不要で、ブラウザで公式ビルドプログラムを開き、ローカルディレクトリを選択するだけで開発を開始できます。すべての計算、データ、ストレージ操作はローカルで実行され、クラウドサービスへの依存はありません。

### AI フレンドリー，検証しやすい

コンパイルのブラックボックスがないため、AI が生成したコードは迅速にデプロイし自己検証できる；中間層を減らし、コンパイルプロセスを経由せず、コードの問題特定と修正を容易にする。

### マイクロフロントエンドのネイティブサポート

ofa.js は、Web フロントエンド開発をマイクロサービスのように、複数の独立したモジュールに分割し、各モジュールを独立して開発、独立してデプロイできるようにします。従来の Web フロントエンドの境界が打破されると、フロントエンド技術は徐々にサーバー技術の制限を突破していくでしょう。
